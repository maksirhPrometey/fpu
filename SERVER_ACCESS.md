# Підключення до сервера ФПУ

Тестовий сайт: **https://testwww.fpsu.org.ua**

## Дані для підключення

| Параметр | Значення |
|---|---|
| Хост | `testwww.fpsu.org.ua` |
| Порт SSH | `9092` |
| Користувач | `root` |
| SSH-ключ | `~/.ssh/fpsu_key` |
| Шлях до проєкту на сервері | `/mnt_data/fpsu` |
| ОС сервера | FreeBSD 13.5 |
| База даних | SQLite (`/mnt_data/fpsu/db.sqlite3`) |

> Ключ `~/.ssh/fpsu_key` (і `fpsu_key.pub`) має бути у вас у папці `~/.ssh/`. Якщо ключа немає — зверніться до попереднього адміністратора, щоб отримати приватний ключ.

## Підключення по SSH

```bash
ssh -p 9092 -i ~/.ssh/fpsu_key root@testwww.fpsu.org.ua
```

Якщо права на файл ключа занадто відкриті, SSH відмовить у підключенні — виправте:

```bash
chmod 600 ~/.ssh/fpsu_key
```

## Копіювання файлів на сервер (SCP)

Завантажити один файл:

```bash
scp -P 9092 -i ~/.ssh/fpsu_key шлях/до/файлу root@testwww.fpsu.org.ua:/mnt_data/fpsu/шлях/до/файлу
```

Приклад — оновити `views.py`:

```bash
scp -P 9092 -i ~/.ssh/fpsu_key apps/core/views.py \
  root@testwww.fpsu.org.ua:/mnt_data/fpsu/apps/core/views.py
```

## Типові команди на сервері

Після підключення по SSH:

```bash
cd /mnt_data/fpsu

# Завантажити змінні середовища й виконати Django-команду
set -a; . .env; set +a
./venv/bin/python manage.py <команда>

# Приклади
./venv/bin/python manage.py migrate
./venv/bin/python manage.py shell
```

### Перезапуск сайту (gunicorn) без падіння (502)

```bash
sh /mnt_data/fpsu/restart_gunicorn.sh
```

Цей скрипт робить «м'який» перезапуск (`kill -HUP`), тому сайт не падає з 502 під час деплою.

## Синхронізація даних з оригінального сайту fpsu.org.ua

Локально в корені проєкту є готовий скрипт — запускається однією командою:

```bash
./sync
```

Він сам:
1. завантажує актуальний `sync_from_live.sh` на сервер;
2. підключається по SSH;
3. запускає імпорт нових новин, оновлення напрямів діяльності, фото керівництва та членських організацій;
4. показує прогрес у терміналі.

## Безпека

- Ніколи не публікуйте приватний ключ `fpsu_key` і вміст `.env` (там паролі, `SECRET_KEY`, дані пошти).
- Не вимикайте `StrictHostKeyChecking` — за потреби просто підтвердьте відбиток ключа сервера під час першого підключення.
- Для нового співробітника краще згенерувати окремий SSH-ключ і додати його публічну частину в `~/.ssh/authorized_keys` на сервері, а не передавати спільний приватний ключ.
